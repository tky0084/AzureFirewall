## 概要

もし、特定の宛先に対して通信ができない場合は、その原因がどこにあるか探ることが大切です。<br>
今回は、その原因特定をAzureFirewallの観点に絞って確認してみましょう。<br>
今回ご紹介する方法は、AzureFirewallの診断設定ログから確認する方法です。

<br>

## ログの確認

それではログを確認してみましょう

作成したAzureFirewallを開き、左ペインから「監視 | ログ」を押下する

<img width="480" height="258" alt="スクリーンショット 2026-05-19 090251" src="https://github.com/user-attachments/assets/1a6d1073-10d3-42c6-9d91-cee24c373f62" />

<br>

クエリを開く画面が出てくるので、「AzureDiagnostics」と入力して実行する

<img width="960" height="669" alt="スクリーンショット 2026-05-19 090521" src="https://github.com/user-attachments/assets/439fb5d0-6d10-4b76-bdf7-87cc1a7bf8a5" />

<br>

これではどの情報を見ればいいのかよくわからないため、必要な列だけ絞り込みます。
この場合、「| project {列名}, {列名}...」という構文を使用します。

```
AzureDiagnostics
| project //以下の必要な列を検索する
    TimeGenerated, //ログの日時
    Action_s, // 許可/拒否の結果
    Category, // 「AZFWNetworkRule」/「AZFWApplicationRule」のどちらか
    SourceIP, // 送信元IPアドレス
    Protocol_s, //ポート番号
    DestinationPort_d, //ポート番号
    DestinationIp_s, //宛先IPアドレス
    Fqdn_s, //宛先FQDN
    Policy_s, //AzureFirewallポリシー名
    RuleCollectionGroup_s, //規則コレクショングループ名
    RuleCollection_s, //ヒットした規則コレクション
    Rule_s, //ヒットしたルール名
    ActionReason_s //主にDenyされた理由が出力される（ルールが存在しないなど）
```

<br>

するとどうでしょうか。<br>
どこからの(SourceIP)、どこに対する(DestinationIp_s / Fqdn_s)、どのような(Protocol_s, DestinationPort_d)通信が、許可/拒否されているか(Action_s)、<br>
またどのルールで許可/拒否されているか(RuleCollectionGroup_s、RuleCollection_s、Rule_s)
拒否されている場合はなぜか(ActionReason_s)が表示されました。

<img width="960" height="416" alt="スクリーンショット 2026-05-19 091319" src="https://github.com/user-attachments/assets/5402e050-8b20-48dd-93ff-48e4590e4657" />

<br>

それでは、Spokeの仮想マシンから、「[www.google.com](http://www.google.com/)」、「[www.yahoo.co.jp](http://www.yahoo.co.jp/)」にアクセスしてみましょう

そして、クエリの2-3行目に、必要なレコード(行)を絞り込むためのwhere構文を追記します。<br>
・2行目は送信元をSpoke環境のIPアドレスに等しいものを指定<br>
・3行目は宛先FQDNが「[www.google.com](http://www.google.com/)」または「[www.yahoo.co.jp](http://www.yahoo.co.jp/)」のいずれかに等しいものを指定

```
AzureDiagnostics
| where SourceIP == "10.0.1.4" 
| where Fqdn_s in ("[www.google.com](http://www.google.com/)", "[www.yahoo.co.jp](http://www.yahoo.co.jp/)")
| project //以下の必要な列を検索する
    TimeGenerated, //ログの日時
    Action_s, // 許可/拒否の結果
    Category, // 「AZFWNetworkRule」/「AZFWApplicationRule」のどちらか
    SourceIP, // 送信元IPアドレス
    Protocol_s, //ポート番号
    DestinationPort_d, //ポート番号
    DestinationIp_s, //宛先IPアドレス
    Fqdn_s, //宛先FQDN
    Policy_s, //AzureFirewallポリシー名
    RuleCollectionGroup_s, //規則コレクショングループ名
    RuleCollection_s, //ヒットした規則コレクション
    Rule_s, //ヒットしたルール名
    ActionReason_s //主にDenyされた理由が出力される（ルールが存在しないなど）
```

すると、穴あけされているルールのログと、穴あけされていないルールのログの違いが判ると思います。<br>
穴あけされている「[www.google.com](http://www.google.com/)」はルール名が表示されてAction_sが「Allow」となっており、<br>
穴あけされていない「[www.yahoo.co.jp](http://www.yahoo.co.jp/)」は「No rule matched」または「Default Action」などと表示されています。

##  現場におけるトラブルシューティングの方法

今回は、AzureFirewallで許可ルールにヒットしたという結果が確認できました。<br>
また、「No rule matched」などが表示されれば、ルールが設定されていないことによるものだということがわかりました。

<br>

では、**何もログが出ていない場合**はどうでしょうか？<br>
上記から、ログが出ていないということは、AzureFirewallが通信ルールを評価していないということになります。<br>
つまり、**その通信はAzureFirewallを通過していない**ということになります。

その場合は通信できない理由は他にあるので、別の観点で調査が必要となります。<br>
※サブネットや仮想マシンのNSG、仮想マシンのMicrosoft Defender Firewall(OSのFirewall)、不正なルートテーブルなど


## 補足

今回のDenyログは<u>穴あけがない状態のもの</u>であり、明示的に拒否するルールを作成した場合、<br>
Action_sはDenyになり、Rule_sにどの拒否ルールで拒否されたかというログが出力されます。

ほかにも絞り込みたいレコードがあれば、where構文で絞り込んでみましょう。
